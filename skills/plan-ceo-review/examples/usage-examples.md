# plan-ceo-review 使用示例

## 示例 1: 绿地项目 — 股票预警系统

### 用户输入

```
我想做一个股票预警系统，当股价突破关键点位时推送通知
```

### 总指挥审核

#### Step 0: 核心范围挑战

**这是正确的问题吗？**

用户说"股价推送"，但真正想要的是：
- ✅ 不错过交易机会
- ✅ 及时知道何时行动
- ❌ 不是"收到一堆通知"

更简单的路径：
- 如果用户只想看特定股票 → 复用 eastmoney-financial-data
- 如果用户想要智能提醒 → 需要新建

**现有代码能复用吗？**

| 子问题 | 现有能力 | 复用方式 |
|--------|---------|---------|
| 获取股价 | eastmoney-financial-data | 直接调用 |
| 实时数据 | qveris-official | 直接调用 |
| 推送通知 | 无 | 需新建 |
| 技术分析 | 无 | 需新建 |

**理想状态轨迹**

```
当前: 无预警系统
  ↓
本次: 基础股价推送
  ↓
12个月理想: AI 投资助手
           - 个性化推荐
           - 可操作建议
           - 自学习优化
```

**模式选择**: EXPANSION（绿地项目）

#### 扩展提案

**10x 检查**

什么版本野心大 10 倍但努力只多 2 倍？

```
基础版: 股价 → 推送
         Effort: S

10x 版: 股价 → 技术分析 → 持仓感知 → 可操作建议 → 推送
         Effort: M（只多 1 级）
         Value: 10x
```

**柏拉图理想**

如果世界最好的工程师有无限时间：

```
用户打开 App → 看到:
  ┌─────────────────────────────────┐
  │ 今日关注                         │
  ├─────────────────────────────────┤
  │ 🔔 茅台突破 1800 关键位           │
  │    建议: 观察                    │
  │    理由: 成交量未放大，可能是假突破 │
  ├─────────────────────────────────┤
  │ 🔔 比亚迪回踩 250 日均线          │
  │    建议: 考虑加仓                 │
  │    理由: 均线支撑 + 新能源板块走强 │
  └─────────────────────────────────┘
```

**惊喜机会（5 个）**

1. 一键执行建议（如果对接券商 API）
2. 历史准确率：过去建议的胜率
3. 社区观点：其他用户怎么看
4. 学习优化：根据用户操作改进
5. 风险提醒：不只是机会，还有风险

#### AskUserQuestion

```
你想要哪个版本？

A) 基础版 — 股价突破推送
   - Effort: S（人类: 2天 / AI: 30分钟）
   - Completeness: 6/10

B) 智能版 — 持仓感知 + 风险偏好 + 可操作建议  ⭐ 推荐
   - Effort: M（人类: 1周 / AI: 2小时）
   - Completeness: 8/10

C) AI 助手版 — 自动分析 + 个性化推荐 + 学习优化
   - Effort: L（人类: 1月 / AI: 1天）
   - Completeness: 10/10
```

## 示例 2: 功能迭代 — 优化新闻分析速度

### 用户输入

```
buy-side-news-analyst 技能运行太慢了，想要优化
```

### 总指挥审核

#### Step 0: 核心范围挑战

**这是正确的问题吗？**

用户说"优化速度"，但可能是：
- ✅ 真的慢 → 优化性能
- ❌ 感觉慢 → 添加进度反馈
- ❌ 等待焦虑 → 添加预估时间

**现有代码能复用吗？**

| 子问题 | 现有能力 | 复用方式 |
|--------|---------|---------|
| 并行处理 | sessions_spawn | 多 Agent 并行 |
| 进度反馈 | 无 | 需添加 |
| 缓存 | 无 | 可添加 |

**模式选择**: SELECTIVE（功能迭代）

#### 樱桃挑选仪式

**扩展机会 1: 添加进度反馈**

```
当前: 黑盒运行，用户不知道进度
改进: 显示当前步骤和预估剩余时间

Effort: S（人类: 2小时 / AI: 5分钟）
Risk: 低
Value: 显著改善用户体验

A) 加入本次范围
B) 延期到 TODOS.md
C) 跳过
```

**扩展机会 2: 并行处理**

```
当前: 串行分析多个新闻源
改进: 多 Agent 并行处理

Effort: M（人类: 1天 / AI: 30分钟）
Risk: 中（需要处理竞态）
Value: 速度提升 3-5x

A) 加入本次范围
B) 延期到 TODOS.md
C) 跳过
```

**扩展机会 3: 结果缓存**

```
当前: 每次重新分析
改进: 缓存相似新闻的分析结果

Effort: S（人类: 4小时 / AI: 15分钟）
Risk: 低
Value: 重复查询秒回

A) 加入本次范围
B) 延期到 TODOS.md
C) 跳过
```

## 示例 3: Bug 修复 — 推送失败无日志

### 用户输入

```
股票预警推送有时会失败，但没有日志，不知道什么原因
```

### 总指挥审核

#### Step 0: 核心范围挑战

**这是正确的问题吗？**

用户说"推送失败无日志"，但可能是：
- ✅ 推送服务不稳定 → 添加重试
- ✅ 没有日志 → 添加日志
- ❌ 被风控拦截 → 可能需要换个推送方式

**模式选择**: HOLD（Bug 修复，不做范围扩展）

#### 7 维审核（简化）

**Error & Rescue Map**

| 方法 | 错误 | 是否救援 | GAP |
|------|------|---------|-----|
| PushService#send | 网络超时 | ❌ | GAP 1 |
| | Token 过期 | ❌ | GAP 2 |
| | 频率限制 | ❌ | GAP 3 |

**Observability**

| 项目 | 当前 | 需要 |
|------|------|------|
| 日志 | 无 | 结构化日志 |
| 指标 | 无 | 成功率、延迟 |
| 告警 | 无 | 失败率 > 5% 告警 |

#### 决策

**接受的范围**：
- [ ] 添加结构化日志（入口、出口、错误）
- [ ] 添加错误救援（超时重试、Token 刷新、频率限制退避）
- [ ] 添加监控指标

**NOT in scope**：
- 更换推送服务（当前问题不够严重）
- 添加推送历史记录（延期）

## 示例 4: 范围削减 — 大型重构

### 用户输入

```
想把整个 Agent 体系的配置系统重构成统一的配置中心
```

### 总指挥审核

#### Step 0: 核心范围挑战

**涉及文件统计**: 23 个文件

**模式选择**: REDUCTION（超过 15 文件）

**无情切割**：

```
原范围:
- 统一配置中心
- 所有 Agent 迁移
- 配置热更新
- 配置版本控制
- 配置审计日志

最小可行版本:
- 统一配置中心核心
- 总指挥 Agent 迁移（验证可行性）
- 基础配置读取

后续 PR:
- 其他 Agent 迁移
- 配置热更新
- 版本控制和审计
```

#### 决策

**接受的范围**：
- [ ] 统一配置中心核心
- [ ] 总指挥 Agent 迁移

**后续 PR**：
- [ ] 其他 Agent 迁移
- [ ] 配置热更新
- [ ] 版本控制
- [ ] 审计日志

# plan-ceo-review

**总指挥专用任务审核框架**

---

## 触发场景

- 收到复杂任务（3+ 步骤）
- 需要做战略决策
- 功能开发前规划
- 方案评审

---

## 核心理念

### Boil the Lake（煮湖原则）

> AI 让完整性成本接近零，所以要完整实现，而不是"差不多就行"

- **湖** = 可以完全做到的事（100% 测试覆盖、完整错误处理、所有边界情况）
- **海** = 做不到的事（整个系统重写、多季度迁移）
- **原则**：煮湖，别煮海

---

## 四种模式

| 模式 | 目标 | 适用场景 | 默认推荐 |
|------|------|---------|---------|
| **EXPANSION** | 扩大范围，找 10 星产品 | 全新功能、产品规划 | ⭐ 绿地项目 |
| **SELECTIVE** | 保持范围 + 展示扩展机会 | 功能迭代、增强 | ⭐ 默认推荐 |
| **HOLD** | 严格审核，不做扩展 | Bug 修复、重构 | ⭐ 紧急任务 |
| **REDUCTION** | 削减到最小可行版本 | 范围过大、时间紧 | ⭐ 超过 15 文件 |

---

## 审核流程

### Step 0: 核心范围挑战（必做）

回答三个问题：

1. **这是正确的问题吗？**
   - 用户真正想要的结果是什么？
   - 有没有更简单的路径？

2. **现有代码能复用吗？**
   - 项目中已有类似功能？
   - 能否组合现有模块？

3. **理想状态是什么？**
   ```
   当前状态 → 本次任务 → 12个月理想
   ```

### Step 1: 选择模式

根据任务类型选择：
- 绿地项目 → EXPANSION
- 功能迭代 → SELECTIVE（默认）
- Bug 修复 → HOLD
- 超过 15 文件 → REDUCTION

### Step 2: 7 维审核

| 维度 | 核心问题 | 输出 |
|------|---------|------|
| **Architecture** | 数据流、状态机、依赖关系 | ASCII 图 |
| **Error Handling** | 每个失败路径都有救援？ | 错误表 |
| **Security** | 输入验证、授权边界？ | 安全清单 |
| **Edge Cases** | nil、空、边界、并发？ | 边界表 |
| **Testing** | 覆盖率、回归测试？ | 测试矩阵 |
| **Observability** | 日志、指标、告警？ | 监控清单 |
| **Deployment** | 回滚、Feature Flag？ | 部署计划 |

### Step 3: 输出到文件

```
findings.md → 审核发现
task_plan.md → 任务计划（阶段化）
progress.md → 进度跟踪
```

## 示例用法

```
用户: 我想做一个股票预警系统，当股价突破关键点位时推送通知

总指挥: 让我审核一下这个任务...

[运行 plan-ceo-review]

## Step 0 分析

**这是正确的问题吗？**
用户想要的是"不错过交易机会"，不是"股价推送"。
更好的方案可能是：
- 智能筛选：只推送用户关注的股票
- 情境感知：结合用户持仓和风险偏好
- 可操作：不只是通知，还要给建议

**模式选择**: EXPANSION（绿地项目）

**理想状态**:
当前: 无 → 本次: 基础预警 → 理想: AI 投资助手

## 扩展提案

1. **基础版**（原需求）
   - 股价突破推送
   - Effort: S

2. **智能版**（推荐）
   - 持仓感知、风险偏好、可操作建议
   - Effort: M → 价值提升 3x

3. **AI 助手版**（10 星）
   - 自动分析、个性化推荐、学习优化
   - Effort: L → 价值提升 10x

你选择哪个版本？
```

---

## 注意事项

1. **不要一次性问太多问题** - 每次只问一个决策点
2. **推荐完整选项** - 基于煮湖原则，优先推荐完整实现
3. **决策要记录** - 所有决策写入 findings.md
4. **模式要一致** - 选定模式后不要静默漂移
